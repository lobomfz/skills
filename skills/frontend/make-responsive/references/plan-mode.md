# Plan Mode

Use this when `make-responsive` receives a markdown plan generated by `mobile-review`.

## Detecting A Plan

Treat input as a plan when it is markdown and includes at least:

- `# Mobile Review`
- `## Context`
- one of `## Direct Fixes`, `## Broad Fixes`, or `## Layout Decisions`

If the shape is close but incomplete, explain what is missing and ask whether to proceed manually.

## Parsing

Extract:

- Target.
- Files read.
- Assumptions.
- Accepted and skipped direct fixes.
- Broad fixes and affected files.
- Layout decisions, chosen options, and rationale.
- Pending items.

Skipped items stay skipped unless the user explicitly says to review them.

## Staleness Check

For every file referenced by accepted work:

1. Read the plan generation date when present.
2. Compare with `git log -1 --format=%ai -- <file>`.
3. If the file changed after the plan, list it as stale in Turn 1.

Stale items do not auto-apply. Put them in the individual decision flow.

## Code Match Check

Before applying each plan item, verify that the described code still exists.

- One clear match: update line numbers silently and apply.
- Multiple matches: list as ambiguous and ask.
- No match: list as stale/missing and ask.

Do not force a plan onto code that no longer matches.

## Turn 1

```text
Read `<plan>` generated on <date>.

Will apply:
- <N> direct fixes
- <N> broad fixes
- <N> complex component decisions

<warnings if stale, ambiguous, or unmappable>

Reply `ok` to apply, or `review X` to reopen specific items.
```

Warnings should be concrete:

```text
Needs review before applying:
- `OrdersTable.tsx`: changed after the plan was generated.
- `UsersTable.tsx`: plan says "make it better on mobile", which does not map to a concrete action.
```

## `review X` Override

The user may reopen:

- a number from the Turn 1 list,
- a file name,
- `review all`,
- `review stale`,
- `review default`.

Apply non-reopened items first, then ask reopened items one at a time. For reopened items, use the plan decision as the recommendation:

```text
**Recommended: rewrite as cards.** This was the decision in the review plan.
```

If the user chooses a different action, note that the plan was overridden in the summary.

## Direct Fixes

Accepted direct fixes apply without showing the table again. If the user asks to see them, show a compact table before applying.

Skipped direct fixes remain unchanged.

## Broad Fixes

Apply broad fixes before individual component decisions. Remove covered child items from the apply queue.

If the broad fix no longer maps cleanly, reopen it as an individual decision.

## Unmappable Decisions

Reopen decisions that are vague or absent from the known action set:

- "make it nicer"
- "improve mobile"
- "follow the screenshot" without a concrete implementation detail

Ask the decision from scratch, using the plan as context but not as authority.

## Empty Plan

If the plan says no work is needed, say so and stop. Do not create a no-op diff.

## Summary

Include:

- Applied plan path.
- Items applied automatically.
- Items reopened and final choices.
- Items skipped.
- Stale or missing items left pending.
